# Fix Date Format in Properties & Strip Date‑Time Prefix from Meeting Name

## 1. Goal

Two targeted fixes to the frontmatter generation logic in `convertTranscript.ts` and its supporting `deriveMeetingName` function:

1. **Date property** — change from `YYYY-MM-DD HH:mm:ss` to date‑only `YYYY-MM-DD`.
2. **Meeting name** — strip any leading date/time prefix from the filename before deriving the human‑readable name (the actual file on disk is never renamed).

All existing tests must be updated accordingly, and overall test coverage must be verified to meet or exceed **80 %**.

***

## 2. Detailed Requirements

### 2.1. Date Property — Date Only

**Current behaviour (line 48 of `convertTranscript.ts`):**

```ts
const dateStr = moment(fileCreationTime).format("YYYY-MM-DD HH:mm:ss");
```

Produces frontmatter like:

```yaml
date: 2026-04-05 14:30:00
```

**Required behaviour:**

```ts
const dateStr = moment(fileCreationTime).format("YYYY-MM-DD");
```

Produces:

```yaml
date: 2026-04-05
```

Only the `date` property in the frontmatter is affected. Dialog‑line timestamps (`convertTxtToMarkdown`, `convertVttToMarkdown`) remain unchanged.

### 2.2. Meeting Name — Remove Date/Time Prefix

Source filenames may contain a leading date/time segment that should not appear in `meeting_name`. Common patterns include but are not limited to:

| Filename (basename)                             | Expected `meeting_name`   |
| ----------------------------------------------- | ------------------------- |
| `2026-04-05_14-30_weekly_sync`                  | `Weekly Sync`             |
| `2026-04-05_weekly_sync`                        | `Weekly Sync`             |
| `20260405_1430_weekly_sync`                     | `Weekly Sync`             |
| `20260405_weekly_sync`                          | `Weekly Sync`             |
| `weekly_sync`                                   | `Weekly Sync`             |
| `meeting_saved_closed_caption`                  | `Untitled Meeting`        |

**Approach — update `deriveMeetingName`:**

1. Before any other processing, attempt to strip a leading date/time prefix from the basename.
2. Recognised prefix patterns (anchored to start of string):
   - `YYYY-MM-DD_HH-mm_`  — e.g. `2026-04-05_14-30_`
   - `YYYY-MM-DD_`         — e.g. `2026-04-05_`
   - `YYYYMMDD_HHmm_`     — e.g. `20260405_1430_`
   - `YYYYMMDD_`           — e.g. `20260405_`
3. A single regex that covers all four variants (order matters — try the longest match first):

   ```ts
   basename = basename.replace(
       /^\d{4}-?\d{2}-?\d{2}(?:_\d{2}-?\d{2})?_/,
       ""
   );
   ```

4. If the strip results in an empty string (the entire basename was a date), fall back to `"Untitled Meeting"`.
5. After stripping, the existing logic applies: check for `meeting_saved_closed_caption`, replace underscores, title‑case.

**Important:** This is purely a display transformation for frontmatter and the `# heading`. The source file is never renamed.

### 2.3. Validation

Because the file name may or may not contain a date prefix, `deriveMeetingName` must handle both cases cleanly:

- If the basename has no recognisable prefix, it is used as‑is (current behaviour preserved).
- If the basename is entirely a date string (e.g. `2026-04-05`), treat it as `"Untitled Meeting"`.
- If stripping the prefix leaves only whitespace or underscores, treat it as `"Untitled Meeting"`.
